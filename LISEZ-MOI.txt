## 🎯 Objet de cette livraison

**Permettre la modification d'une commande déjà envoyée**, tant qu'elle n'est
pas encore passée en statut "Commandée".

### Le problème terrain identifié

Aujourd'hui : un monteur passe une commande, oublie un article → il en passe
une nouvelle. Résultat : 2-3 petites commandes au lieu d'une grosse, ce qui
encombre l'historique et complique le travail des achats.

### La solution

Bouton **"✏️ Modifier"** sur la fiche d'une commande, visible si :
- Statut ∈ {En attente de validation, Validée, Envoyée aux achats}
- Utilisateur = demandeur de la commande OU Admin/Direction

L'écran de panier s'ouvre pré-rempli avec les articles de la commande, le user
modifie, sauvegarde. Les anciens articles deviennent les nouveaux. Aucune
nouvelle commande n'est créée.

---

## ⚙️ Comportement détaillé

### Statut après édition

- Si le demandeur a `directAchat: true` (Admin / Direction)
  → **Le statut est conservé** (pas de revalidation requise)
- Si le demandeur a `directAchat: false` (Monteurs / Chefs / Conducteurs sans flag)
  → **La commande repasse en "En attente de validation"** automatiquement
  → Le conducteur du chantier devra revalider

### Notification Direction si statut était "Envoyée aux achats"

Si la commande modifiée était déjà envoyée aux achats par email, une entrée
est créée dans la collection Firestore `notificationsPending` avec un message
prêt-à-envoyer pour la Direction. Comme le système SMS Brevo n'est pas encore
branché côté code (à brancher en v10.H ou v10.I via Make), pour l'instant
l'avertissement passe par :

1. Un toast à l'écran lors de la modification : "✏️ Commande modifiée — la
   Direction est notifiée (achats à re-prévenir)"
2. Une bannière sur l'écran de récap pendant l'édition
3. Un document dans `notificationsPending` que Make pourra traiter une fois
   branché à Brevo

### Trace des modifications

Chaque édition ajoute une entrée dans le tableau `editHistory[]` du document
Firestore de la commande. Format :

```javascript
{
  by: "Joseph BILARDO",
  at: "2026-05-12T14:30:00Z",
  previousStatut: "Envoyée aux achats",
  newStatut: "En attente de validation",
  summary: "2 articles ajoutés, 1 qté modifiée"
}
```

L'historique s'affiche dans une carte dédiée sur la fiche commande (dans
l'historique des commandes), avec le format :
```
📝 Historique des modifications (2)
• 12/05/2026 14:30 — Joseph BILARDO : 2 articles ajoutés
  statut : Envoyée aux achats → En attente de validation
• 12/05/2026 15:00 — Mickael COURTEAU : Date de réception
  statut : En attente de validation → En attente de validation
```

### Champs modifiables

| Champ | Modifiable ? |

## 🐛 Risques identifiés et atténuations

### 1. Concurrence : 2 personnes éditent la même commande
- **Risque** : la dernière sauvegarde gagne, l'autre perd ses modifs
- **Atténuation** : aucune dans cette livraison. En pratique, EPJ étant une
  petite équipe, ce cas devrait être rare. À reprendre si besoin avec un
  système de "verrou" Firestore.

### 2. SMS Direction non branché
- **Risque** : la Direction n'est PAS notifiée via SMS quand une commande
  envoyée est modifiée
- **Atténuation** : la notification est enregistrée dans
  `notificationsPending` avec tous les éléments nécessaires. Le toast
  pendant l'édition affirme que "la Direction est notifiée" — techniquement
  c'est vrai (la queue Firestore est remplie), mais le SMS lui-même ne part
  pas pour l'instant. À brancher en v10.H/I via Make + Brevo.

### 3. Annulation au milieu d'une édition
- **Risque** : l'utilisateur clique "✏️ Modifier", change le panier, puis
  ferme l'app. Le panier garde l'état pour la prochaine session, mais
  `editingOrder` est dans un useState (perdu au rechargement)
- **Atténuation** : au prochain démarrage, le draft localStorage restaure le
  panier, mais on est dans un état "création nouvelle commande" pas "édition".
  Conséquence : si l'utilisateur "Envoie" cette commande, il créera un doublon.
  À corriger en v10.G.3 si ça pose problème : sauvegarder editingOrder dans
  localStorage aussi.

### 4. Suppression d'un article qui n'existe plus dans le catalogue
- **Risque** : si une commande contient un article qui a été supprimé du
  catalogue depuis, on l'affiche quand même (snapshot dans la commande)
- **Atténuation** : l'item est conservé dans le panier pendant l'édition.
  Pas de problème.

### 5. Statut "Validée" éditable
- **Risque** : si un conducteur a validé, et que le demandeur réedite, la
  commande repasse en "En attente". Le conducteur perd sa validation.
- **Atténuation** : c'est le comportement voulu. La nouvelle version doit
  être revalidée. L'historique trace qui a fait quoi.

### 6. Email supplémentaire intentionnel modifié
- **Risque** : un user pourrait éditer pour ajouter son email perso et
  recevoir une copie du PDF lors de l'envoi aux achats
- **Atténuation** : faible risque (déjà existant en mode création), trace
  dans editHistory.

### 7. Quantité passée à 0
- **Risque** : si l'utilisateur met 0 dans le panier, l'article est-il
  retiré ?
- **Atténuation** : `cartItems` filtre déjà sur `q > 0` (filter), donc oui
  l'article sort de la commande. Comportement cohérent avec la création.

### 8. Édition d'une commande "Envoyée aux achats" puis envoi mail
- **Risque** : la Direction modifie la commande. Statut reste "Envoyée aux
  achats". Mais le mail aux achats a été envoyé avec l'ancienne version.
- **Atténuation** : la trace `notificationsPending` est créée, qui rappelle
  que la version aux achats est obsolète. La Direction sait qu'elle doit
  renvoyer le mail (manuellement pour l'instant, ou via Make plus tard).
